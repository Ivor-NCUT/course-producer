# 行业专家知识萃取访谈

只设计访谈与记录协议，不代替专家回答，也不直接写课程正文。

开始前读取定位、`materials.jsonl`、缺口报告，以及 `知识库/方法论/专家访谈.md` 和 `知识库/模板/访谈记录模板.md`。

## 先判断是否需要访谈

把课程蓝图所需信息与现有证据逐项对照。若现有材料已经覆盖关键判断、步骤、案例、边界和讲师语言，输出“无需补访谈”的证据清单和剩余非阻塞缺口，把阶段完成为 `no_action` 产物；不要重复问已有答案。

## 设计动态问题

每个问题对象必须包含：

- `question_id` 与它要填补的 `gap_id`；
- 主问题：只问一件事，指向真实经历或决策；
- 已知信息：引用已有 `source_id`，让提问不从零开始；
- 追问条件：根据模糊回答、缺少例子、缺少边界、出现矛盾等信号触发；
- 证据请求：数字来源、原文件、截图、时间、角色或可复盘案例；
- 完成判定：拿到什么信息就停止追问。

问题组合应覆盖：

1. 学员的认知起点与常见错误判断；
2. 希望建立的认知终点；
3. 具体挑战情境，包括人物、环境、事件、问题和约束；
4. 专家的判断标准、操作步骤、失败边界和例外；
5. 可讲的案例、故事、数据与高价值原话；
6. 学员可执行的行动和成果验收。

不要一次机械问完题库。访谈执行时一次问一个主问题，根据回答选择追问；已经达到完成判定就进入下一缺口。

## 输出

生成 `.course-producer/artifacts/interview-guide.md`：

- 访谈目标与已有材料摘要；
- 按优先级排列的问题卡；
- 每张卡的条件追问、证据请求与结束判定；
- 可跳过的问题及其已知证据；
- 整场访谈结束条件；
- 使用 `访谈记录模板.md` 的记录约定。

## 记录分类

访谈记录中的每个条目只能归入一种类型：

- `fact`：可外部核查的事实或数据；
- `opinion`：专家判断、解释或价值立场；
- `case`：含角色、情境、行动和结果的经历；
- `claim_to_verify`：尚无证据的事实性主张；
- `ceo_decision`：需要负责人选择课程方向或承诺的决定。

每条记录保留 `question_id`、时间戳或段落位置和原话；不要把观点改写成事实，也不要把待验证主张直接写进课程承诺。

## 结束判定

整场访谈在以下条件同时满足时结束：所有 P0 缺口有答案或明确无法回答；关键方法至少有一个完整案例；高风险事实有证据或转为待验证；课程承诺和边界没有未决冲突。剩余低优先级问题写进缺口清单，不用拖长访谈。

完成后把访谈记录交给 `course-knowledge-extraction`。
